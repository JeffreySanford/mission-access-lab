## 2. Storybook — done

- [x] Scaffolded via `nx g @nx/storybook:configuration access-portal` (`@storybook/angular` 10.5.3, Nx's inferred-plugin path — no `angular.json` needed). This took real work to get running; three genuine blockers in order:
  1. **`@storybook/angular`'s webpack builder now hard-requires the Angular Architect builder path** (`SB_FRAMEWORK_ANGULAR_0001`) whenever `@angular-devkit/build-angular` is resolvable on disk — and it always is, because `@nx/angular` (used for its ESLint config) declares it as a peer dependency that pnpm auto-installs regardless of what's in our own `package.json`. The fix: don't fight it — define explicit `storybook`/`build-storybook` targets in `apps/access-portal/project.json` using the real Angular Architect builders (`@storybook/angular:start-storybook` / `:build-storybook`) with `browserTarget: "access-portal:build:development"`. Nx executors are interoperable with Angular Architect builders, so no `angular.json` is required — Nx's own inferred `@nx/storybook/plugin` targets (which invoke the plain, no-longer-supported `storybook build` CLI) are simply overridden by these explicit ones.
  2. **Compodoc** is on by default for both builders and isn't installed — set `"compodoc": false` on both targets (not needed for this lab).
  3. **Duplicate webpack instances**: `@storybook/builder-webpack5` and `@angular-devkit/build-angular` each resolve `webpack@5.106.2` with different peer contexts under pnpm's default strict isolation, producing two separately-instantiated copies of the "same" version — `instanceof Compilation` checks then fail across the plugin boundary (`TypeError: The 'compilation' argument must be an instance of Compilation`). A `pnpm.overrides` pin on the version alone doesn't collapse peer-qualified duplicates. Fixed by switching to `node-linker=hoisted` in `.npmrc` — a workspace-wide change, but verified the full `verify:release` suite (lint/test/build, not just Storybook) still passes after it.
  4. Also needed `apps/access-portal/tsconfig.json` (the generator expects the standard Nx per-project tsconfig; this workspace's app/spec tsconfigs extended the base directly without one).

- [x] Negative space, two cases — and one of them **found and fixed a real bug**: `OpenFgaAuthorizationAdapter` never caught exceptions from the OpenFGA HTTP call, so a malformed object string (which OpenFGA correctly rejects with `400 Bad Request`) propagated uncaught all the way to a generic `500` with the misleading message "wrapper service unavailable" — wrong on two counts: not a 5xx-class problem, and not actually a service-availability issue. Fixed by catching `HttpClientErrorException` (4xx — fail closed, "OpenFGA rejected the request: ...") and `RestClientException` (5xx/connection failure — fail closed, "OpenFGA is unavailable: ...") separately, both defaulting to DENY rather than throwing. Updated `OpenFgaAuthorizationAdapterTest`'s two exception-propagation tests to assert the new fail-closed decisions instead (they previously documented the broken behavior as "known, not faked" — now it's fixed instead). Also covered: a cross-organization user denied access.

Wrapped in try/finally for best-effort cleanup.
- [x] Infra-restart negative-space pass — done, as its own dedicated suite (`apps/access-portal/e2e/resilience/`, run via `pnpm run e2e:resilience` / `nx run access-portal:e2e-resilience`), deliberately **not** part of `access-portal:e2e`/`verify:release` (each case force-recreates the wrapper container or stops OpenFGA — 30-60s per restart, riskier against the single shared dev stack than the golden-path suite is worth gating every commit on). All three cases pass (4 tests, ~2m50s):

Also empirically confirmed (contrary to the initial assumption) that Spring Boot 4.1's OAuth2 resource-server JWT decoder resolves the issuer's JWKS metadata *lazily*, not at startup — the wrapper boots fine even though no `mission-access` realm is actually provisioned in the Keycloak container, and a request with no bearer token correctly gets `401` without ever needing to reach Keycloak; `/actuator/health` correctly stays open (`permitAll` on the health/info endpoints).

## 4. Security review — done

This is explicitly a training lab, not production — decided what "secure" means here rather than writing tests against an undefined target.

- [x] `SECURITY_ENABLED` defaults to `false` (`apps/authorization-wrapper/src/main/resources/application.yml`) — decision: keep the default off, documented loudly in a comment right above the property. Flipping it on would require wiring real Keycloak token issuance into the Angular playground and the E2E suite (the deferred negative-space case in section 3 already tracks the "missing JWT" scenario); doing that just to change a default wasn't worth the churn for a lab whose whole point is demonstrating the OpenFGA ReBAC decision path, not authn. `SecurityConfiguration` already enforces the real `oauth2ResourceServer` JWT path correctly when `SECURITY_ENABLED=true` is set — verified by reading the filter chain, not assumed.
- [x] `infra/.env` — confirmed gitignored (`.gitignore:13`) and never committed. Added `infra/.env.example` documenting the two keys it holds (`OPENFGA_STORE_ID`/`OPENFGA_MODEL_ID`), noting they're populated automatically by `pnpm run fga:bootstrap`.
- [x] Dependency CVE sweep (`pnpm audit`) — found 18 vulnerabilities (2 high, 15 moderate, 1 low), all in transitive dev-tooling (Nx-internal `axios`/`brace-expansion`, `webpack-dev-server` via `@nx/webpack`, `esbuild` via `@angular/build`'s vite, `@hono/node-server` via `@storybook/angular`'s CLI dep) — none reach the deployed application bundle. Pinned safe patched versions via `pnpm.overrides` in `package.json` (`webpack`, `axios`, `brace-expansion`, `esbuild`, `@hono/node-server`, `webpack-dev-server`, `uuid`, plus a nested `minimatch@3.1.5>brace-expansion` pin). `pnpm audit` now reports "No known vulnerabilities found"; `pnpm run verify:release` passes clean with the overrides in place (verified locally end-to-end, and CI run `29943345451` is green on the commit that includes them).
- [x] CORS configuration — no CORS config exists anywhere in `apps/authorization-wrapper` (no `@CrossOrigin`, `CorsConfigurationSource`, or `.cors(...)` on the Spring Security chain), so Spring Security's default (deny cross-origin unless explicitly allowed) applies. Not a gap: `apps/access-portal/proxy.conf.json` proxies `/api` and `/actuator` from the Angular dev server (`:4200`) through to `:8080` server-side, so the browser never makes a cross-origin request to the backend in the documented dev flow — CORS headers are never invoked. If this backend were ever called directly from a different origin, it would be rejected by default rather than exposed by a permissive misconfiguration.
- [x] SQL injection surface — audited. The only repository, `AuthorizationAuditRepository`, is a plain `JpaRepository` with no custom `@Query` methods, no derived-query strings, no `JdbcTemplate`/`EntityManager`/native queries anywhere in the module. The Flyway migration is static DDL. No hand-built SQL/JPQL exists in the codebase — no injection surface found.
- [x] Secrets in `infra/compose.yaml` (Postgres/Keycloak admin creds) — confirmed intentional lab-grade plaintext; added a comment at the top of the file stating this explicitly so it doesn't read as an accidentally-shipped "real" pattern.
